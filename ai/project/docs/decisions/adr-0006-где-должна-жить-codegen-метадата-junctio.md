# ADR-0006: Где Должна Жить Codegen Метадата Junctio

**Date:** 2026-07-22
**Source:** Discussion #13

## Context

Проблема: codegen-метадата `junction` жила в YAML-**ключе** `.spy.yaml` — пространстве
имён строгого валидатора Serverpod, который её отвергает (`property is not allowed`).
Директива была нерабочей: пользователь снимал её руками перед каждым `serverpod generate`.

Ответ на главный вопрос (Q4 — нужна ли директива вообще): **нужна**. Проверены 5
структурных сигналов (порядок relation-полей, blanket-exclude ownership-FK, unique-индекс,
имя класса, back-relation) — каждый ломается на реальных данных. Два relation-поля с
идентичной структурой не содержат признака «ownership vs junction-parent»; `customerId`
в `TaskTagMap` — tenant-scope, в `CustomerUser` — настоящий родитель. Директива остаётся
как escape hatch на неоднозначность.

Решение: сменить не форму, а **носитель** — из YAML-ключа в YAML-комментарий
(`# codegen:junction: [a, b]`). Это устраняет корень (вторжение в чужой namespace), а не
симптом, и не требует правок ни в одной из точек вызова Serverpod. Механизм уже
существует в проекте (`# manifest:` в 16 файлах).

Fallback ужесточается отдельным этапом после миграции шаблонов, чтобы не ломать
`create-project` из коробки. Каскад-эвристика отклонена как решение с остаточным
silent-риском.

**Артефакты:** TASK-040 (носитель C) → задача на ужесточение fallback (depends_on).

## Decision

**Принято владельцем 2026-07-22.** Второе мнение teamlead'а: ключевые факты дискуссии
перепроверены по коду и файлам (16 `.spy.yaml` с `# manifest:` — подтверждено;
`marker_analyzer.ts:35` читает `#`-комментарии — подтверждено; контрпример GPT_2 про
required attribute-FK — подтверждён на реальном `weight/customer_user.spy.yaml`, где
`defaultTerminalSetId: UuidValue?` тривиально теряет `?`).

### 1. Носитель — вариант C (comment-directive)

`# codegen:junction: [a, b]` в том же `*_map.spy.yaml`. Плоский namespaced-маркер,
якорь на колонку 0, чтение до `yaml.load`. Синтаксис пары `[a, b]` не меняется.

Обоснование: Serverpod физически не видит комментарий → все 13 известных точек вызова
(и неперечислимый хвост: пользователь руками по DoD, IDE-плагин Serverpod, CI) зелёные
без единой правки. Носитель уже обкатан в этом же репозитории — `# manifest:` живёт в
16 шаблонных `.spy.yaml` и ни разу не дал бага.

**Отклонены:** B (враппер+strip) — чинит 13 известных вызовов, оставляет неперечислимый
хвост, мутирует пользовательский файл; A-sidecar — принят как резервный, но платит двумя
файлами за чистоту, которую репо осознанно не соблюдает; A-CLI-флаг — не персистится,
допустим только как разовый override.

### 2. Обе формы директивы мигрируют

`junction: [a, b]` (TASK-037) И `junction: true` (TASK-013) болеют одинаково — Serverpod
отвергает **ключ**, а не значение. Чинить обе, иначе баг вернётся через `junction: true`.

### 3. Fallback — компромисс GPT с усилением: каскад НЕ внедряется

Порядок работ (каждый шаг обратно совместим, регрессии нет ни на одном):

1. **TASK-040:** только носитель C + чтение обеих форм из комментария + migration-guard
   на старый YAML-ключ. **Fallback остаётся ТЕКУЩИЙ.** Ничего не ломается.
2. **Владелец (другие репо):** миграция шаблонов t115/simplified + строка в weight.
   Директива-комментарий безвредна для старого кода, поэтому делается заранее.
   Тест-фикстуры внутри репо кодогенератора — в п.1 или отдельной задачей.
3. **Отдельная задача (заводится СРАЗУ, не «потом»):** ужесточение fallback → fail-fast
   при структурной неоднозначности, с `depends_on` на п.2 и явным условием старта —
   только после подтверждения владельца, что шаблоны мигрированы.

**Каскад (не-nullable + имя ∉ {customerId, userId}) НЕ внедряется вообще.** Причины
владельца: даёт третье состояние fallback (эвристика → каскад → fail-fast), которое
придётся выпиливать; оставляет silent-путь — контрпример с required attribute-FK реален
и подтверждён на файле.

### 4. Общая конвенция

`codegen:<key>: <value>` — единый синтаксис codegen-директив, общий с решением
дискуссии #14 (`// codegen:ownership:`). Не разводить два похожих синтаксиса.

### 5. Снято выбором варианта C (НЕ «отклонено по существу»)

> ⚠ Формулировка уточнена владельцем (см. `## Approved`): это вопросы, **снятые
> конкретным выбором носителя**, а не закрытые по существу.

- **Единый враппер вызова `serverpod generate`** — при варианте C **не требуется**
  (Serverpod не видит комментарий, чинить нечего). Остаётся полезной гигиеной (логи,
  флаги, версия) отдельной задачей. **Если когда-нибудь вернётся подход B (strip) —
  враппер снова становится ОБЯЗАТЕЛЬНЫМ:** без него strip в одной точке оставит
  остальные падать.
- **Требование unique-индекса у junction** — при варианте C не нужно как источник пары.
  Остаётся открытой темой целостности БД само по себе.
