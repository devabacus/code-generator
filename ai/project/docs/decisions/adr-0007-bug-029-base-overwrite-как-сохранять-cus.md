# ADR-0007: Bug 029 Base Overwrite Как Сохранять Cus

**Date:** 2026-07-22
**Source:** Discussion #14

## Context

Дискуссия началась с вопроса «как сохранять custom-код в `:base`» и закончилась
переворотом постановки: `:base` оказался **защищённой** частью (merge работает, код вне
блока выживает), а беззащитны 65 из 81 шаблона, идущие через полную замену файла без
единой проверки. Проблема в 7 раз шире исходной формулировки, и большая её часть не была
заведена.

Решение — не новые маркеры, а **guard на границе записи**: двухфазный fail-closed preflight
плюс ledger хешей машинного вывода как baseline, позволяющий отличить «пользователь правил
файл» от «шаблон обновился». Оба компонента неделимы: preflight без ledger нежизнеспособен
из-за prompt fatigue, ledger без preflight ничего не предотвращает.

Отклонены per-method markers и patch-only. Вариант с partial-файлами снят как основанный
на неточной посылке (sync hooks не используют partial). Правка тел generated-методов
честно вынесена за скобки как нерешаемая маркерами.

**Артефакты:** BUG-029 расширяется (было/стало зафиксировано) → задача на preflight+ledger.

## Decision

**Принято владельцем 2026-07-22.** Второе мнение teamlead'а: ключевые факты перепроверены
по коду. Подтверждено независимо: **81 entity-шаблон, 16 с `:base`, 65 без** (числа
Claude_1 — 74/9 — неточны в общем количестве, но ключевое «65 беззащитных» совпало ровно);

> **Как считалось** (критерий обязателен — иначе число невоспроизводимо; расхождение
> 74 vs 81 = разные критерии подсчёта). Замер 2026-07-22 по `G:/Templates/flutter/t115`:
>
> ```bash
> T="G:/Templates/flutter/t115"
> grep -rl "manifest: entity" "$T" | wc -l                                    # → 81 (всего)
> grep -rl "manifest: entity" "$T" | xargs grep -l "generated_start:base" | wc -l  # → 16 (с :base)
> # 81 − 16 = 65 (без :base → СТРАТЕГИЯ 2, полная замена)
> ```
>
> Критерий: файл считается entity-шаблоном, если содержит маркер `manifest: entity`
> (в любой форме комментария). Подсчёт рекурсивный по всему дереву шаблона, включая
> server/admin/flutter части.
СТРАТЕГИЯ 2 (`generation_service.ts:221-239`) вызывает `createFile()` без проверки
существования и без предупреждения; файлы обрабатываются через `Promise.all` — guard
внутри `_processFile` не остановит конкурентные записи.

### 1. Переформулировка BUG-029 (подтверждена владельцем)

**Было:** «`:base` секции перезаписываются при regenerate».
**Стало:** «65 из 81 entity-шаблона идут через `createFile()` без проверки существования».

`:base` — это как раз **защищённая** часть: механизм merge существует, работает, и код
вне блока переживает regen (`_mergeBaseContent` заменяет только область между маркерами).
Беззащитно всё остальное — включая ровно те файлы, которые пользователь правит руками:
`repository_impl`, `usecases`, `entity`, `model`, `state_providers`, `endpoint`, адаптеры.

**Bug-report: расширяется существующий BUG-029, новый НЕ заводится** (один класс дефекта —
потеря пользовательского кода при regen; два номера про одно уже проходили с BUG-005).
В теле обязательна явная фиксация «было / стало / дата / ссылка на эту дискуссию».

### 2. Три режима silent-отказа (все обязаны стать громкими)

1. **Silent loss:** existing-файл без `:base` заменяется целиком (65 файлов).
2. **Silent loss внутри owned-region:** пользователь изменил `:base`, контракт не виден
   и ничем не защищён.
3. **Silent staleness:** шаблон содержит `:base`, а target потерял/сломал маркеры —
   `_mergeBaseContent` возвращает исходный target без ошибки, файл навсегда перестаёт
   получать обновления.

### 3. Первый deliverable — preflight + ledger ВМЕСТЕ (неделимая единица)

Guard без baseline мёртв: каждый штатный regen (добавили поле → изменился вывод во всех
~21 файле сущности) упирался бы в conflict → пользователь рефлекторно жмёт
`--overwrite-existing` → guard перестаёт защищать ровно тогда, когда custom-код есть.

**Три технических инварианта (обязательны в контракте):**

- **(а)** Preflight выполняется **ДО начала любой записи**, не внутри `_processFile`:
  там `Promise.all`, бросок в одном promise не остановит остальные → полузаписанное дерево.
  Поток двухфазный: plan (вычислить все destination + новый контент, классифицировать,
  не трогая FS) → apply (только если конфликтов нет или получено явное подтверждение).
- **(б)** Для merge-файлов хешируются **регионы**, не файл целиком. Иначе легальное
  добавление custom-импорта/метода в preserved-зону даст conflict на пустом месте.
- **(в)** Legacy-состояние **НЕ «усыновляется»** как generated. Если existing ≠ render и
  пользователь выбрал «оставить как есть» — нельзя записывать хеш existing как baseline
  (внутри может быть custom-код → на следующем regen `existing == ledger` → молчаливое
  стирание). Допустимо только: reviewed overwrite (записать render, хешировать render)
  либо preserve (не писать и не seed'ить).

Дополнительно из дискуссии: ledger (`.codegen/ledger.json`) versioned, в git,
project-relative пути, точный SHA-256 без lossy-нормализации, коммитится в FS
**последним** — после успешного apply (атомарной заменой временного файла).

### 4. Отклонено

- **Вариант 1 (per-method markers)** — решает задачу, которую уже решает merge-стратегия,
  ценой N маркеров на файл и хрупкой привязки к именам методов.
- **Вариант 2 (patch-only как постоянный режим)** — перекладывает merge на пользователя.
  Diff из preflight полезен как интерфейс подтверждения, но не как режим работы.
- **Вариант 3 в исходной формулировке («partial-файлы как sync hooks»)** — опирался на
  неточность в постановке вопроса: проверено, sync hooks живут в **том же файле** после
  маркеров, никаких пользовательских partial-файлов нет. Плюс Dart не поддерживает partial
  class. Реально обкатанный паттерн — «узкие marker-блоки + остальной файл territory
  пользователя»; он и распространяется на merge-файлы.

### 5. Вынесено за скобки (отдельные вопросы)

- **Правка тела** сгенерённого метода (напр. добавить фильтр в `getTasks`) не сохраняется
  **ни одной** маркерной схемой. Это вопрос точек расширения (callback/strategy/wrapper/
  subclass), отдельное API-решение — не делать вид, что маркеры его закрывают.
- Миграция 65 шаблонов на merge-дисциплину — инкрементально, по файлу, после guard'а.
- Явный ownership-контракт (`// codegen:ownership: generated|merge`) — отдельной строкой-
  директивой, НЕ через перегрузку `manifest:` (`marker_analyzer.ts:39` делает слепой каст
  токенов в `ManifestType` без валидации).

### 6. Stack-lock

Preflight, ledger, ownership-комментарии и перестановка marker-комментариев **не задевают**
stack-lock (Riverpod annotations, Drift, Clean layout, sync_core 0.3.0 не меняются).
Partial-файлы / наследование / новые callback API — задевают, требуют явного approval.
